# R1-C · 沟通效率协议 — 多项目并行下「人 ↔ agent ↔ 多 harness」的沟通设计

> 三路独立设计之「沟通效率」路。不含技术适配层（事件怎么采集）、不含大盘 UI（驾驶舱长什么样），只回答一个问题：**什么消息、在什么时刻、走什么通道、以什么形态到达王佐，其余一概静默。**
>
> 第一性：沟通的本质 = 让正确的事在对的时刻到达人。推论三条，全文档只从这三条推导：
> 1. **王佐的注意力是唯一稀缺资源**，每条通知都在花这笔钱——默认不发，发必须挣回成本。
> 2. **通道即语义**：飞书 = "现在需要你";H5 角标 = "有空看"；日报 = "存档+回看"。事件选对通道，协议就成功了一半。
> 3. **每条消息必须可行动或纯档案**，中间态（"告诉你一声"）是垃圾。

## 1. 通知分级协议

### 1.1 四级定义

| 级 | 名字 | 语义 | 通道 | 到达形态 |
|---|---|---|---|---|
| **P0** | 立即打断 | 现在不看会出真事 | 飞书即时单条消息（触发手机系统通知） | 单卡，带动作按钮 |
| **P1** | 聚合推送 | 今天需要处理，但不差这几十分钟 | 飞书聚合卡片，按节律批送 | 一张卡装多条，逐条可点 |
| **P2** | 静默待阅 | 值得知道，不值得打断 | 不推飞书；进 H5 收件箱角标 + 次日日报 | 角标数字 + 日报条目 |
| **P3** | 档案 | 只作数据源 | 只进事件库 | 不产生任何提醒 |

### 1.2 事件 → 等级统一映射表（跨 harness 唯一真源）

所有 harness(Kimi / Claude Code / Codex / fleet headless）上报同一套事件类型，等级只由下表决定，**harness 无权自定等级**:

| 事件 | 条件 | 等级 |
|---|---|---|
| 审批请求 | 命中红线（动钱>¥1000 / 生产写入 / 凭据 / 对外不可逆发布 / 首次对客交付） | **P0** |
| 审批请求 | 普通可逆操作，阻塞会话继续 | **P1** |
| 提问 | 阻塞型（不答就停），且 agent 已给出建议选项 | **P1** |
| 提问 | 无建议无选项的开放提问 | **不许发**（退回 agent 补建议，见 §5) |
| 任务失败 | 终态失败，人已重试不了、必须介入 | **P1** |
| 任务失败 | 自动重试后成功 | **P2**（只留"曾失败已自愈"一行） |
| 任务完成 | 交付物需王佐过目/验收（发客户成稿等） | **P1**（走过目流，带成品 URL) |
| 任务完成 | 无需验收的常规完成 | **P2** |
| 额度告警 | 用量 ≥80% | **P1**;≥95% 且影响在途任务 → **P0** |
| 机器/harness 离线 | 有在途任务被卡住 | **P1**；卡住的是 P0 审批 → **P0** |
| 离线 | 无在途任务 | **P2** |
| 开始工作 / 进行中 / 心跳 / cron 成功 | — | **P3**（永不提醒） |

### 1.3 防轰炸规则（硬机制，不靠自觉）

- **单会话冷却**：同一会话同类事件 10 分钟内只出一条，后续合并计数("及另外 2 项")。
- **同类合并**：同一聚合窗内多条 P1 合成一张卡："3 个会话等你拍板 / 1 个失败待处理"，绝不逐条刷。
- **P1 批送节律**：每 30 分钟攒一批；攒够 5 条立即送；工作时间（9:00–23:00）外不批送（入早报）。
- **夜间静默 23:00–8:00**：仅 P0 中的红线/生产事故可破静默；其余全部攒进次日早报。
- **全局预算**:P0 每周 >5 条说明分级失灵或系统有病，触发协议复盘；P1 目标 ≤6 条/天（见 §6)。
- **幂等去重**：每条事件带幂等键，跨 harness 重复上报只算一次。
- **升级有限**:P1 超时未决只重提醒一次（见 §2.3)，之后执行该项预设的保守默认，不无限轰炸。

## 2. 拍板/审批协议（决策闭环）

### 2.1 闭环链路

```
agent 产生决策需求（带建议+利弊+选项，缺一不上抛）
  → 决策服务单写入账（幂等键，状态=pending）
  → 按级触达:
      P0 → 飞书单卡(按钮可直接拍)
      P1 → 飞书聚合卡(简单项按钮直拍;复杂项深链)
  → 王佐拍板:
      ① 飞书卡片按钮(1 击,二选一/三选一场景)
      ② 深链进 H5 拍板台(多项/需上下文,白底蓝按钮标准台)
  → collector 落账(append-only,末次为准)
  → 回写原会话(结果注入,agent 继续执行)
  → 飞书发最终回执("已按【认可】执行/已记录")——回执不依赖 H5 页面还开着
```

三条铁律（承继既有资产，不重造）:
- 拍板台一律走 `interactive-decision-h5` 标准技能生成（白底蓝按钮 + collector 回流），**禁止手搓展示页冒充拍板台**。
- 每个决策项必须自解释 + 平民化术语 + 带总控明确建议；缺建议的退回，不上王佐。
- 展示与决策物理分离：状态看板里不放提交区，只放"→去拍板台"深链。

### 2.2 时效与升级

- 每个决策项创建时**强制带**:`deadline` + `超时默认动作`。超时默认必须保守（不执行/维持现状/取风险最低项）;**红线事项的超时默认永远 = 不执行**，不存在"超时自动放行"。
- 提醒节奏：创建即推（按级）→ 时效过半未决，聚合重提**一次** → 到期执行默认动作 + 飞书回执告知"已超时按默认处理"。
- P0 不受此限:P0 超时未决每 2 小时重推，直到有人拍或事件失效（红线的语义就是"必须人来")。

### 2.3 批量模式

- 例行多事项（如每日待拍清单）走 **digest 变体**：一张卡/一个页，逐项「认可/要改」+ 一次性提交全部 + 末尾补充说明卡（纯文本选填，不再逼整体二拍）。
- 同主题多会话的审批自动归并为一组，不按会话拆散——王佐拍的是"事"，不是"会话"。

## 3. 日报/周报协议

### 3.1 日报（每天 8:30，一条飞书卡片 + H5 全文深链）

硬上限结构，超出进 H5 二级页：

1. **【需你拍板】≤3 件** — 每条带按钮/深链。这是日报存在的核心理由。
2. **【昨日交付】≤5 条** — 只列"完成了且对你有意义的"，一句话一条，带产物链接；中间过程不写。
3. **【异常与风险】≤3 条** — 失败未愈/额度/离线，每条带"建议动作"。
4. **一行舰队数字** — 活跃会话 / 完成 / 失败 / 额度水位，一行内。

**可行动条目占比 ≥50%**：日报里至少一半内容是可点可拍的；纯状态汇报超一半是日报的失败。王佐在等的过目件就绪即推，**禁压进次日早报**。

### 3.2 周报（周一 9:00)

- 项目维度，每项目三行：本周进展 / 下周计划 / 卡点（卡点必须带建议）。
- 决策回顾：本周拍板次数、平均响应时长、超时默认执行了几次（校准协议用）。
- 无按钮无拍板，纯回顾 + 一个"展开看项目"深链。

### 3.3 生成方法

- 数据源 = 全部 harness 上报的统一事件流（含完成/失败/审批/额度）+ plan-SSOT 挂账。
- 生成 = 聚合查询 + 模板渲染，**不允许 agent 自由发挥写日报文案**（防变成又一种噪音）；只有"异常与风险"的建议动作由 agent 填，且按决策分诊标准自查。
- 王佐未打开日报连续 3 天 → 日报自动降级为纯 H5 存档（证明了"不该发"，协议自己纠偏）。

## 4. 人在回路的最小摩擦：交互模式清单与选型

| 模式 | 动作数 | 适用 | 结论 |
|---|---|---|---|
| **A. 飞书交互卡片按钮**（按钮回调直写 collector) | 1 击 | 二选一/三选一：认可/否决、批准/拒绝、选 A/B/C | **默认首选**，覆盖 ~80% 拍板 |
| **B. 飞书深链 → H5 拍板台** | 2 击（点链接→逐项拍） | 多项批量、需要看上下文/产物/附件 | 复杂决策标准路径，链接必须 https+域名（飞书内置浏览器限制） |
| **C. 飞书直接回复指令**（回 "认可 3" / 引用回复） | 打字 | 按钮失效/网络降级时 | **兜底通道**，解析失败必须回执确认，永不静默吞指令 |
| **D. kimi-remote 会话页处理**（pending_interaction 徽标） | ≥3 步 | 必须看会话上下文才能定的审批 | 只用于此类；能从卡片说清的绝不引流到会话页 |

选型原则：**能在卡片里说清并拍掉的，绝不让王佐跳转**。卡片自解释（是什么/让你拍什么/拍了怎样/总控建议）是 A 模式成立的前提；说不清的才升级 B。C 是可靠性兜底，D 是上下文兜底。

## 5. 反模式：永远不许发的消息

1. **"我开始工作了"/"进行中"/心跳** — 零信息量，P3 都嫌多，只进事件库。
2. **cron/定时任务成功通知** — 成功是默认态，只有失败值得说。
3. **无建议无选项的开放提问**("你看怎么办？")— 退回 agent 补建议+利弊+选项再来。
4. **同一事件第 3 次提醒** — 重提一次是上限，之后走超时默认。
5. **纯状态汇报出日报之外**("今日运行正常")— 正常不值得一条消息。
6. **含内部代号黑话的消息**(M5/SSOT/collector…)— 文案闸拦截。
7. **23:00–8:00 的非红线消息** — 静默期铁律。
8. **元通知**("通知系统已更新/即将发日报")— 系统不谈论自己。
9. **可自决事项的上抛** — 违反决策分诊：技术/机制/已授权事项自决+一句话简报（P2)，不做卡片不推飞书。

## 6. 验证方式（一周试运行）

### 6.1 上线前

- 映射表（§1.2）落地为一份机器可执行的配置文件，所有上报事件强制过表定级；模拟 50 条历史事件回放，人工核对定级无误判红线漏 P0。

### 6.2 试运行指标（每条通知落账：等级/发出时间/打开时间/处理动作/处理时长）

| 指标 | 目标 | 判读 |
|---|---|---|
| P0 条数 | ≤2/天，≤5/周 | 超 = 分级失灵或 fleet 真有病，复盘 |
| P1 条数 | ≤6/天 | 超 = 合并/冷却规则失效 |
| 通知打开率 | ≥90% | 发了没人点 = 这条不该发，逐条回溯 |
| P1 中位处理时长 | <30 分钟（醒着时段） | 长 = 卡片不自解释或选型错（该用 B 用了 A) |
| 超时进入默认执行数 | 红线项 = 0；普通项 <10% | 红线超时 = 协议事故 |
| 误报率（王佐标"不该发") | <10% | 逐条校准映射表 |
| 漏报数（事后发现"该告诉我没说") | 0 | 与误报率对冲，防过滤太狠 |

### 6.3 每周校准会（15 分钟）

抽样 20 条已发通知 + 全部 P3 事件清单，逐条问三个问题：该不该发？该发哪级？该走哪个交互模式？答案回写映射表（§1.2）与选型表（§4)。**协议的正确性不靠设计保证，靠这张表每周被真实数据修正。**

---

## 7. 与另两路的边界

- 本协议只定义"事件→等级→通道→形态→闭环"的规则与文案标准；事件的采集/上报管道（技术适配层）与收件箱/看板 UI（大盘路）按本协议消费同一份定级结果。
- 净减少承诺：本协议落地后应退役的旧通道 = 各 harness 各自的零散 notify 脚本直推（收敛到唯一定级出口）；不新增任何"通知中心"类新前端（复用 kimi-remote 角标 + 标准拍板台）。
