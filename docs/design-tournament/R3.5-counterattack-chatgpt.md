## 0. 反击结论(先行)

红蓝判**对**的是:「人工导入快照」该死——owner 原话「我永远不会手动导入」,蓝队 X5 命中。
红蓝判**错**的是:把「人工导入」当成 chatgpt 接入的唯一形态。R1-A §2.4 与 R2 评审只评估了
「share-link + 手动粘贴」一条数据源,**从未评估过全自动数据源**。穷尽候选后存在一条
全自动、合规成本可控、边际成本远低于红队估计的组合:

**E(官方 Codex CLI 读 cloud 任务)+ B(Playwright 网页快照,spike 门控)+ D/F(ingest 兜底)**,
三层全部只读、全部 snapshotOnly,无任何 recurring 手工操作。且脱敏管线在 v0.5 本就交付
(VERSION-PLAN v0.5 含「分享链接(只读+脱敏管线)」,owner 反馈溯源),chatgpt 是搭车不是拖车。

**排期结论:chatgpt 接入进 v0.5(骨架+E+D+B-spike)/ v0.6(B 实现,spike 通过才做),
反对「整条线推迟到 P3 后」。**

## 2. 候选评估

格式:可行性(证据)/ 合规与账号风险 / 数据新鲜度 / 维护成本 / 结论。

### A. 桌面 App 本地缓存解析(conversations-v3 *.data 逆向) — **否决**

- 可行性:差。实测缓存只有 1 个会话文件、二进制无明文(grep 零命中);2025-03 曾有
  「ChatGPT macOS 明文存储聊天记录」报道([acudc.com](https://acudc.com/chatgpt-macos-users-surprised-by-unencrypted-chat-storage/)),
  当前 26.715 已非明文(推断为收紧后的序列化缓存)。格式私有无文档,社区无公开解析器(检索无果)。
  即使逆向成功,拿到的也只是「最近打开的会话」缓存,无枚举全量能力。
- 合规:读本机文件本身合规,但依赖私有格式,版本升级即失效。
- 新鲜度:看似实时,实则不可控(App 何时写/清缓存不可知)。
- 维护:高——对私有二进制格式的永久跑步机。
- 结论:**否决**。登记 needs_discovery,若未来出现开源解析器再议。

### B. ChatGPT 网页版自动化(Playwright 定期静默快照) — **有条件采纳(spike 门 + 可弃层)**

- 可行性:中高。chatgpt.com 在 Cloudflare 后;2025-2026 的 CF/Playwright 对抗指南仍在活跃更新
  ([Browserless 2026-01](https://www.browserless.io/blog/bypass-cloudflare-with-playwright)、
  [Apify 2025-08](https://blog.apify.com/bypass-cloudflare/)),共识是「真实浏览器 + persistent
  session + 低频人类节奏」可通过,但这是军备赛,无永久保证。本机 Playwright + Chrome for Testing 已装。
  两个子模式诚实分级:
  - **B1 DOM 读取**(推荐):渲染后读侧边栏列表与会话正文。最贴近「用户自己看」,合规面最好;脆弱于 UI 漂移。
  - B2 借浏览器会话取 backend-api JSON:稳定但等同程序化调私有 API,灰色加深,不作默认。
  Profile 策略:独立 user-data-dir(落 adapter state 目录)+ **一次性登录**(一次性授权 ≠ recurring
  手动导入,owner 红线是后者)。备选 CDP attach 日常 Chrome 需改启动参数,侵入日常使用,不作默认。
- 合规:灰。OpenAI 消费级条款对自动化提取有限制(通用认知;spike 任务含逐字核对最新条款文本)。
  本方坚持:只读、低频、本人数据、**永不碰控制**(红线原样)。实践风险 = 挑战/强制重登,小概率账号警告。
  **此项需 owner 在 decisions.md 显式签收;B 设独立配置开关,一键退役不影响其他层。**
- 新鲜度:轮询级(列表 5-15min 增量,正文按需拉取)。对「看」的场景足够;它不是控制面,不需要秒级。
- 维护:中。选择器漂移 + 登录态维护;用 kill-switch 封顶投入(§4 死因 3)。
- 结论:**采纳为全覆盖主力,但 spike 先行**(v0.5 限时 0.5-1 天出 probe 报告:登录态、挑战率、列表 DOM 稳定性),v0.6 才实现,退役条件预写。

### C. 官方 API(platform.openai.com API key 拉会话) — **否决**

- 可行性:对本需求 = 零。`/v1/conversations` **存在**([OpenAI API reference](https://developers.openai.com/api/reference/resources/conversations/methods/update/),conv_* 资源),
  但那是 **Responses API 平台侧**会话对象;消费端 ChatGPT(App/网页)会话与平台 API 账号体系**不互通**,
  API key 与 Pro 订阅是两套。即:端点存在,但里面没有 owner 的 ChatGPT 会话。
- 结论:**否决**(数据源不存在,非风险问题)。若 owner 未来用 API 侧 agent,另开需求。

### D. Share-link 抓取 — **降级为 ingest 通道(采纳进组合)**

- 可行性:高(公开链接 SSR 可取,无需登录态),但**无法枚举会话** → 不能作发现源。
- 合规:公开页面,六候选中最干净。
- 新鲜度:仅链接产生时更新。
- 维护:低。且 v0.5 本就交付分享链接管线,chatgpt 侧直接复用同一 ingest 路径,边际 ≈ 0。
- 结论:不作独立方案;作为组合的 ingest 通道之一(别人分享来的会话、点名单个会话)。

### E. Codex cloud 侧(经 Codex CLI 官方表面) — **采纳(合规主力)**

- 可行性:高且有官方支撑。`codex cloud list --json` / `codex cloud status <task_id>` 是官方 CLI 表面
  ([openai/codex#23853](https://github.com/openai/codex/issues/23853) 证实其存在与缺口;
  [#13338](https://github.com/openai/codex/issues/13338)、[#18566](https://github.com/openai/codex/issues/18566)
  证实 ChatGPT 登录态下 `codex cloud list` 可用)。本机 ~/.codex/auth.json 在位。
  ChatGPT App 内的 Codex 任务 = cloud 任务,同账号经 CLI 可读。
  已知缺口:code review 类任务不进 list(#23853);任务正文/日志细度有限(#24777 的 logs/output 仍是提案)
  → 消息粒度诚实 best-effort,不伪造正文。
- 合规:六候选中最干净——官方 CLI + 本人登录态,零逆向零抓取。
- 新鲜度:轮询级;任务状态真实(running/completed/failed)。
- 维护:低(CLI 输出契约远比 DOM 稳定)。
- 结论:**采纳**。定位:chatgpt adapter 的 `origin: codex-cloud` 源——owner 在 ChatGPT App 里看到的
  Codex 任务与聊天同属一个产品面,大盘按产品面分组符合「看」的直觉;且让 chatgpt adapter 从 v0.5
  第一天就有自动数据源,直接拆解蓝队 X5「永远为空」。(备选:并入 codex adapter 加 origin 标签,
  两案差异仅在归组,评审任选。)

### F. 手动粘贴导出(纯前端) — **降级为地板兜底**

- owner 明言永不手动 → 不能作为方案。但作为 ingest 表的纯前端入口成本 ≈ 0,保留为最后一层
  与 B 死亡时的逃生口。红蓝把它当「兜底 = 全部」,本方案把它降回「兜底 = 地板」。

### 补充快速否决

- **G. macOS Accessibility 抓桌面 App**:需 App 可见 + 辅助功能权限,只能抓当前屏,否决。
- **H. 官方数据导出**(Settings → Export → 邮件 zip):全量但需人工触发 + 收邮件,与「永不手动」冲突;
  仅作文档化的一次性历史回填选项,不进方案。

## 3. 最优组合与 adapter 对接设计

「三层快照源」,全部只读、全部 `snapshotOnly=true`:

```
chatgpt adapter(M5,adapter server 内,loopback)
 ├─ E 源 codex-cloud:codex cloud list --json 轮询(1-5min)+ status 按需      [官方/合规地基]
 ├─ B 源 web-chat:Playwright persistent profile,列表 5-15min 增量,正文按需   [spike 门控/可弃层]
 └─ D·F ingest 表(本地,0600):share-link 抓取(复用 v0.5 管线)+ H5 粘贴        [兜底]
        ↓ 归一化为 R1-A §2.1 模型
adapter server 出站统一出口 → §5.3 脱敏过滤器 → 隧道/网关(网关零解析不变)
```

对接 R1-A §2.2:

- **probe()**:三项检查(codex CLI 在位且已登录 / Playwright profile 能无挑战加载会话列表 / ingest 表可写)。
  CapabilitySet:`{readSessions:true, readMessages:true, liveStatus:false, send/interrupt/approval/setMode/upload/nativePush:false}`;
  `conformance='L1'`;`snapshotOnly=true`。
  **分层降级**:B 死 E 在,adapter 不死——`degraded_reason ∈ {'cloudflare_challenge','web_auth_expired','cli_missing','no_local_api', null}`,
  哪个源死报哪个,不装死不 hang。
- **listSessions()**:三源合并。`sid='m5/chatgpt/<nativeId>'`;建议 §2.1 SessionSummary 增补可选字段
  `origin: 'web-chat'|'codex-cloud'|'share-link'|'paste'`(对 R1-A 的小修订,需回写并注明 R3.5 出处)。
  status:codex-cloud 映射真态(running→'running'、failed→'error'、completed→'idle',confidence 'exact');
  web-chat 一律 'unknown',updatedAt 新鲜则 confidence 'heuristic',否则 'stale'(§2.6 规则沿用)。
- **getMessages(sid, cursor)**:快照语义,seq = adapter 内部分配单调序号(对齐 R3 终裁 §6 游标修正),
  epoch = adapter bootId。codex-cloud 正文以 CLI 实际输出为限,缺的部分诚实为空。
- **subscribe()**:轮询 diff 合成 session_event(新会话/updatedAt 变化/任务状态迁移),
  与 §2.5 tail 模拟对 H5 完全同构,H5 断线重连零改动。
- **写路径**:不存在。网关 allowlist 2.0 对 `/m/m5/h/chatgpt/*` 非 GET 双重 403(R1-A §4.1 已设计,沿用)。
- **「用的时候方便灵活」**(owner 后半句):H5 会话详情放「在本机打开」深链——web 会话给
  `https://chatgpt.com/c/<id>`;codex-cloud 任务给任务页深链;桌面 App 注册的 `codex://` scheme
  (实测 Info.plist)作 native 跳转备选,spike 时实测可用性。

**脱敏接入点(§5.3)**:原文只存设备(ingest 表/正文缓存 0600,删除即真删);adapter server 出站统一
出口挂脱敏过滤器——标题→派生标签 `gpt·<short-hash>`(reveal 需显式动作 + 审计),正文→secret 扫描
(redaction 计数进元数据),snapshotOnly 会话 member 默认不可见。管线与 v0.5 分享链接**同一份代码**,
红队「chatgpt 拖着整条脱敏管线」的成本归因不成立。

## 4. 自对抗:本方案最可能死的三种方式

**死因 1:Cloudflare / 登录态杀死 B。** CF 升级指纹、session 过期、挑战率上升;owner 被迫重新登录
= 变相手工操作,恰好复现红蓝预言。
对策:① B 从架构上就是可弃层,E+D 保证 adapter 永远非空;② **spike 门**:v0.5 挑战不可解则不实现 B,
直接以 E+D+F 形态交付并记录理由;③ probe 检测挑战/失登 → degraded_reason 显式上报;④ 人过挑战
逃生口:H5 提示 + 深链回本机浏览器,点一次过完挑战 adapter 自动恢复(一次性点击,非导入);
⑤ 低频 + 真实 Chrome for Testing + 人类节奏降被检概率。

**死因 2:ToS / 账号风险。** 消费级条款限制自动化提取,Pro 账号是 owner 的生产工具,低概率也不可轻忽。
对策:① 只读 + 低频 + 本人数据 + 永不碰控制(红线原样);② B 独立开关一键退役,不影响 E;
③ 风险全文写入 decisions.md **由 owner 显式签收**——这是需要 owner 拍板项,不替他决定;
④ E(官方 CLI)是永久合规地基,B 全死方案仍成立。

**死因 3:价值证伪(红蓝原判在 B 层重演)。** 若 owner 的 ChatGPT 使用主要是闲聊,工作会话本就在
kimi/claude/codex,则 B 的选择器漂移维护再次不成比例。
对策:① **kill-switch 前置写死**:B 上线 4 周内 chatgpt 会话详情打开 < 5 次 → 退役 B,留 E+D+F
(E 成本 ≈ 0 永久保留);② B 工时上限 2-3 天,不做防御性抽象;③ 每个 minor 版复盘看真实使用数据,
用数据裁决而非立场。
附注(第 4 隐患,不单列):E 的覆盖缺口——review 任务不进 cloud list(#23853),owner 若不用 Codex
cloud 则 E 也为空;属死因 3 子情形,由同一 kill-switch 与诚实 degraded 语义覆盖。

## 5. 排期结论(对 VERSION-PLAN 的修订建议)

**明确建议:chatgpt 接入进 v0.5 / v0.6,反对「整条线推迟到 P3 后」。**

- **v0.5(只读联邦)**:chatgpt adapter 骨架(probe/degraded/ingest 表)+ E 源(0.5-1 天)+ D 源
  (复用分享链接管线)+ **B-spike**(限时 0.5-1 天,产出 probe 报告:登录态/挑战率/DOM 稳定性/CLI 路径实测)。
  验收:带 JWT 经公网 `curl /m/m5/h/chatgpt/api/v1/sessions` 返回符合 schema 的清单
  (E+D 至少一源非空,或 degraded_reason 正确暴露);含假密钥/敏感标题 fixture 出站后无原文;
  非 GET 请求网关 + adapter 双重 403。
- **v0.6(可控面)**:B 层实现(**spike 通过才做**,上限 2-3 天)+ F 粘贴入口 + reveal 审计挂钩。
- **退让程序兜底**:若 spike 失败 → chatgpt adapter 以 E+D+F 形态留 v0.5 交付,B 记「为什么不做的理由」
  进 decisions.md(满足反击机制 §2)。
- 修订 VERSION-PLAN v0.5/v0.6 行与 MULTI-HARNESS-DESIGN §3 chatgpt 行,注明本文件出处(反击机制 §2 要求)。

## 6. 对红蓝原判的逐条回应

- **红队「人工导入 = 价值密度最低」**:判对了「人工导入」该死,但数据源枚举窄了——E/B 全自动源在
  R1/R2 完全未被评估(红队 §3 chatgpt 条只评了人工导入形态)。
- **红队「拖着整条脱敏管线」**:成本归因错误。v0.5 本就交付分享链接 + 脱敏管线(owner 反馈溯源),
  管线不依赖 chatgpt 存在;chatgpt 边际 = 过滤器挂钩 + ingest 表。
- **红队「share-link 凭证形态与 CF 对抗未解」**:依然成立——所以 D 只作 ingest 通道不作发现源,
  B 带 spike 门与退役条件。本方案没有假装风险消失,而是把风险关进门和开关里。
- **蓝队 X5「永远不会导入,adapter 永远为空」**:成立的是「人工导入形态」;本方案 adapter 从 v0.5
  第一天就有自动数据源(E),「永远为空」不成立。
- **蓝队「会用两周然后死掉:chatgpt 快照导入」**:对 B 层可能仍对——所以预写 kill-switch,
  用 4 周真实打开数据裁决,而不是现在用立场裁决。
