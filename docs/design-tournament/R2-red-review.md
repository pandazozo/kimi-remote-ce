## 1. 致命(不修则设计不成立)

**[矛盾] C§1.2 × B§1.2 · 「P0」一词两义,等级体系没有唯一真源 · 致命**
B 的 P0 = 阻塞型五件套(待审批/待回答/失败/僵死/红线,§1.2);C 的 P0 = 仅红线审批 + 额度≥95% + 卡住 P0 审批的离线(§1.2),普通阻塞审批在 C 里只是 P1(30 分钟攒一批)。直接后果:大盘(B)上按阻塞时长排第一的"需要你"条目,通知协议(C)不配打断,飞书一片安静;反过来 C 的额度 P0 在 B 的 9 态机里根本没有对应态,手机响了大盘山找不到事。两份文档都自称"唯一真源/分级语义",但语义表本身是两份。
· 建议改法:以 B §1.2 的阻塞分级为唯一事件→等级表,C 降级为该表的「通道与频控」消费者;额度/离线等 C 独有事件类型回灌进 B 的状态机或异常区,一张表维护(物理上就是一个 mapping 配置文件,两文档引用同一份)。

**[矛盾] C§1.2/§2.1 × A§3.2 · 定级引擎与事件汇聚管道没有架构归属 · 致命**
C 要求"所有 harness 上报同一套事件流,过统一映射表定级"(§1.2),拍板要"回写原会话注入结果"(§2.1);但 A 的混合拓扑(§3.2/§3.3)数据面是**拉取式**隧道直连、网关零存储零解析正文,控制面只有机器级心跳。于是:谁 subscribe 全部会话的事件流?定级引擎跑在哪?跑网关 = 违背 A 的 D-11 零解析;跑各机 adapter = 每台机一份定级配置,"唯一真源"破功;跑独立服务 = 就是 A 决策矩阵里被一票否决的中心事件库(②)。拍板闭环依赖的 collector、飞书按钮回调 webhook(公网 https、签名校验、app secret 保管)在 A 的组件清单里完全不存在。
· 建议改法:显式新增一个「notifier/事件汇聚器」组件并写进 A 的拓扑图:跑在**设备侧**(跟 adapter 同机),订阅本机 adapter 事件流,持唯一定级配置,出站只发脱敏后的通知负载;collector 复用 zaios interactive-decision-h5 资产但必须在 A §5 凭证分层表里补一行(飞书 app secret 放哪、回调怎么验签)。这是对 A 拓扑的真实修正,不是文案补丁。

**[安全] C§1.1/§1.2 × B§2.3 × A§5 · 通知内容穿透"凭证不出机"的安全叙事 · 致命**
A 的全部安全设计收敛于"会话正文不出机、网关零存储"(§0.4/§3.2);但 C 的 P0 定义就是"飞书即时单条消息触发手机系统通知"(§1.1),映射表里的审批请求、提问、任务完成摘要(B§2.3 还规定行内显示"审批内容/问题摘要/失败原因一句大白话")意味着:工具参数(可能含路径=客户项目名、命令行=内网地址)、会话标题(B 反模式 13 自己承认标题含过客户密码片段,是"真实踩过的坑")将被推送到①飞书第三方云端留存、②手机锁屏/通知中心上屏。A 把网关失陷的爆炸半径算得很细,却对"内容经通知渠道主动出域"零设防;C 全文唯一的内容管制是反模式 6(内部代号文案闸)。
· 建议改法:通知负载分级脱敏写进 C §1:卡片只放「机器 · 项目 · 事件类别 · 脱敏标题 + 深链」,审批参数/问题正文永远只在 H5(JWT 之后)展示;锁屏通知文案统一为无内容版("你有 1 条待拍板");飞书卡片按钮仅承载决策 id 不承载参数原文。这条规则应与 A §5.3 脱敏管线同源维护。

**[可行性] A§2.5 × A§2.4 · 统一游标「seq = 文件字节偏移」与 codex fork 族自相矛盾 · 致命**
§2.5 规定 jsonl 类 adapter 的 seq 直接取文件字节偏移"天然单调且可恢复";但 §2.1 里 seq 是**会话内**单调,§2.2 的 EventCursor 是**每会话单值** {seq, epoch};而 §2.4 自己实测承认 codex 存在"同 UUID 前缀的多文件 fork 族,游标必须以文件为单位"。多文件组成一个会话时,跨文件字节偏移不可比、无法给出单一单调 seq——P1 验证脚本("杀进程重启→从游标恢复零重复;inode 轮换→不丢行")在 fork 族 fixture 下必然失败。附带:inode 在 APFS 上复用常见,"inode 变化视为轮转"存在漏检窗口;"gap 标记"(§2.4)是 resync_required 之外的新协议元素,违背 §2.5"不发明新协议"。
· 建议改法:对外 seq 改为 adapter 内部分配的单调序号(内存计数 + 持久化 checkpoint),内部仍按文件维护 {inode,offset} 集合,二者解耦;fork 族语义无解时,会话 id 直接以文件为单位(一个 fork = 一个会话),诚实暴露而不是发明合并逻辑;gap 不打标记,直接发 resync_required 复用现有语义。**这是三个候选里最先翻车的机制**,因为它在 P1 就要交付且验证脚本自带反例。

**[安全] A§4.1/§5.1 × decisions D-04 · adapter server 在共享生产机上零认证,现状反而被降级 · 致命**
§4.1 让 adapter server 取代 kimi server 成为隧道唯一上游;§5.1 凭证分层表里 adapter 行只写"绑 127.0.0.1、仅隧道可达",**没有给 adapter 自身定义任何应用层认证**;machine token 是 P3 产物(§6)。而隧道远端端口(127.0.0.1:58627)在 opc-prod 上——D-04 明确该机与 OPC 生产同机、多容器共存。现状该端口由 kimi server 的 Bearer token(KIMI_TOKEN,见 gateway/src/proxy.js:17 注入)挡着;P1 换成 adapter 后若不带等效认证,同机任何本地进程/容器 → loopback 隧道口 → Mac 上全部 harness 控制面(含 claude 写路径、codex exec 臂)。凭证下沉撤掉了静态 .env 泄露向量,却开了一个运行时零认证洞。
· 建议改法:P1 起 adapter server 必须要求 machine token(或最低限度复用 KIMI_TOKEN 作为 adapter 级 bearer),把"adapter 认证"从 P3 提前到 P1 验收标准;§5.1 隧道层与 adapter 层各补一行认证形式。

## 2. 重要(不修则里程碑/指标失真)

**[可行性] A§6-P1 × TEAM-ROADMAP v0.3 · P1 实际掺入了机器命名空间,2 周五路贯通不现实 · 重要**
P1 验证要求 curl `/m/m5/h/{...}/api/v1/sessions` 五路过公网网关——这要求网关已有 machine→upstream 路由,而现状 gateway 是单上游硬编码(proxy.js upstreamBase() 全局单值;ws.js upstreamWsUrl 写死 /api/v1/ws)。机器命名空间 + 注册表在 TEAM-ROADMAP 里是 v0.3 的范围,被悄悄前置进了"2 周"的 P1;WS 桥按 /m/:machine/h/:harness/ 路由与鉴权的改造 A 全文未提。再加:5 个 adapter(kimi 包装、claude SDK 冷读、codex tailer、zcode tailer+未知 schema 归一、workbuddy db+探针)+ allowlist 2.0 + H5 分组。单 owner 2 周做不完;真实体量 4–6 周。
· 建议改法:P1 砍为「adapter server + kimi/claude/codex 三个读路径 + 单机路由(harness 命名空间先行、machine 维度先用单例 m5 硬编码)」;zcode/workbuddy 降为 P1 的 best-effort 不验收;机器维度留 P3 真做。

**[可行性] A§2.3/§6-P1 · workbuddy 探针结果被提前写进验收,「探针后才能定」被写死 · 重要**
开放问题 2 明说"probe 出可验证消息接口前 conformance 封顶 L1",§2.4 说"API 表面未文档化";但 P1 验证脚本要求五路**均返回符合 schema 的清单**。探针失败 = P1 不过——把探索性工作的结果当成承诺。同类:zcode"复用 JsonlTailer"写死了 schema 可归一化的假设(§2.4 只说 best-effort + raw 透传)。
· 建议改法:P1 验收对 workbuddy/zcode 改为「probe 报告产出 + degraded_reason 正确暴露」,清单返回作为通过条件的可选项;把"探针→升级"的门(federation plan 的 live probe gate)显式链接进验收。

**[可行性] A§2.4-claude · 常驻进程回收后,审批恢复路径不存在 · 重要**
每会话一个 streaming-input query() 进程,30 分钟空闲回收、LRU 上限 8;但手机审批的典型场景是"睡前收到、早起批"——8 小时后进程已回收,H5 上 [批准] 按钮点的 resolveApproval 注入到哪?冷会话重新 spawn + resume 审批上下文的路径全文未写。另:§4.3 说 chats.ts"直接迁移不重写",同条又要求"去 HTTP 层、SSE 重放改 session_event 游标重放"——这就是重写,工时按"迁移"估是失真。
· 建议改法:补一条冷恢复设计(resolveApproval 时进程不在 → resume 会话 + 重放 pending approval);P2 验证脚本加"进程回收后审批仍可达"用例;工时按"改造"重估。

**[可行性/安全] A§2.4-codex · mtime<30s 的 TUI 活跃判定会制造它声称要避免的并发写 · 重要**
"TUI 活跃 → 拒绝写"是为了防 exec 与 TUI 并发写同一 thread(锁语义未定义,文档自知);但 TUI 挂着 30 秒无操作 mtime 就不更新 → 判"不活跃" → exec 介入 → 恰好在 TUI 仍开着的情况下写 thread。启发式的失败模式正好是它要防的事故。另外 exec 臂"新消息=一次 exec 全上下文重放"的延迟/成本无预算,P2 验证("确认无写副作用")不测交互可用性。
· 建议改法:TUI 活跃判定改为「进程存活检测(session 文件的 pid 字段/锁文件)+ mtime」双条件,宁可错锁不可错放;exec 臂补延迟预算(如 P95 首 token < X s)进 P2 验证;超预算则该路径诚实降级为"只读 + 深链回本机"。

**[安全] A§5.1 · 「网关失陷损失上限降回路由层被拒」论断夸大 · 重要**
KIMI_TOKEN 下沉只消除了"服务器 .env 静态泄露"一个向量;网关运行时失陷(RCE)仍持有 JWT secret + machine token 表,攻击者经隧道向 adapter 发任意请求,claude 写路径、codex exec 四档(含"完全放行→无沙箱")照打——爆炸半径与现状几乎相同。§5.1 的收尾论断会让评审者误以为下沉解决了运行时失陷。
· 建议改法:§5.1 改成两个向量分开陈述(静态泄露 ✔ 收敛;运行时失陷 → 半径不变,靠 allowlist 2.0 能力位 + exec 沙箱默认只读收敛);exec 臂"完全放行"档要求二次确认或仅本机可设。

**[安全] A§5.3 · chatgpt 脱敏管线只防凭证形态,召回无指标 · 重要**
正文脱敏 = secret 正则(AKIA/sk-/PRIVATE KEY/token 形态)。已知漏网:GitHub PAT(ghp_)、飞书 webhook URL、base64/分段密钥、非标形态 token,以及更常见的业务敏感(客户名、内部代号、报价)——后者 B 反模式 13 在标题上踩过坑,正文反而只有密钥扫描。"规则集与 federation plan redaction policy 同源"引用的是外部文档,本仓无法验证其存在与质量。
· 建议改法:脱敏分两层写明边界(凭证形态 best-effort,不承诺召回;业务敏感靠「snapshotOnly 会话默认 member 不可见 + reveal 审计」的访问控制兜底);给管线定一个 fixture 召回率门槛(如 20 条已知形态 ≥95%)进 P2 验证。

**[安全] A§5.2/§6-P3 · 多账号授权模型两个洞:机器注册无反抢注;WS 订阅无路径级授权 · 重要**
member 一行 curl 装 agent 即上报 MACHINE_ID 注册——没有 admin 审批/归属绑定/重复注册处理,member 可抢注他人 machine id 劫持路由。WS 桥现状只验 JWT、不认路径(ws.js upgrade 仅查 pathname==='/ws'),多机多账号后"member A 订阅 member B 机器的会话流"没有任何文档级别的拦截设计。
· 建议改法:注册表加 (machine_id → owner_user, machine_token) 绑定,重复注册需 admin 确认;WS upgrade 按 /m/:machine 做归属鉴权,P3 验证补"member A 对 member B 机器的 REST **和 WS** 均 403"。

**[矛盾] B§3.2 × A§2.1 · 9 态中 4 态在适配层模型里没有数据源 · 重要**
① `done_unread` 依赖"终态 receipt 事件 + 已读游标":A 的 5 对象模型无 receipt 概念,codex/zcode 的 jsonl 是消息流,**进程退出码/任务终态不在其中**(A§2.6 只能靠"末事件类型非终止集"猜);② `queued` 依赖"prompt 入队事件":TUI 本地输入的 prompt 在 jsonl 里出现即已开始,入队态不可观测,A 模型也无 queue 对象;③ `stalled` 依赖"有未完成任务"字段:A 的 SessionSummary 没有;④ `failed` 的"非零退出"同上无源。同时 B 的 9 态**没有 awaiting_input**——agent 输出一段后等用户回话是 LLM 会话常态,它"有未完成任务 + 最后事件超 TTL" → 30 分钟后全部被打成 `stalled` 假 P0,与 B §1.2 自己警告的"误报率训练用户无视大盘"直接冲突。
· 建议改法:B 状态机与 A §2.1 做一次逐态映射表(每态标注:判据字段、A 是否已提供、缺则补进 A 的 SessionSummary/Message);`stalled` 判据加「非 awaiting_input 态」前置条件;对 heuristic 级状态的 harness,9 态收缩为可判子集并显式标注。

**[矛盾] B§3.1/§4 × A§3.2 · B 要状态存储,A 一票否决了中心事件库 · 重要**
B 要求"状态迁移带时间戳入账"(§3.1,阻塞时长排序与 Today/Week 视图的数据源)、已读游标"服务端化"(§3.3)、Away 摘要要"服务端记录的上次打开时间"(§4.1);A 的混合拓扑网关零存储、不建中心事件库(§3.2 决策矩阵对②一票否决),adapter 侧只有 jsonl 游标持久化。这些状态存到哪、哪个组件提供查询 API(A§2.2 接口里没有),两份文档都没回答。
· 建议改法:明确「状态/已读/迁移历史」的归属:推荐设备侧 adapter 附带一个本地 state store(随 A 的 adapter state 目录),网关只做聚合透传;或承认大盘需要一个元数据级中心存储并修改 A §3.2 的否决范围(正文仍不中心化,状态入账例外)。两案任选,但必须写进文档。

**[矛盾] B§1.2/§3.2 × C§1.3 · 阻塞时长计时起点与通知入账口径分叉 · 重要**
B 的排序键=阻塞时长,起点=状态迁移入账时刻;C 的 P1 批送 30 分钟攒一批、工作时间外压到次日(§1.3)。若 B 的数据管道实时而 C 的通知滞后,王佐次日 8:30 在早报看到"阻塞 9h25m"的条目才首次被通知;若共用同一入账时间,则阻塞时长系统性少算批送延迟。再叠加 jsonl 轮询间隔与机器 30s 心跳,存在三个时钟。另:B 的 stalled TTL=30min(承 SPEC-multisession)与 A 的机器心跳 30s 是两个量级,④异常区的"机器离线"与会话"僵死"用哪套 TTL 未澄清。
· 建议改法:统一规定「一切计时起点 = 事件原始时间戳(adapter 观测点打 ts),通知入账只影响触达不影响计时」; stalled TTL 与机器离线 TTL 分开命名进 B §3.2。

**[矛盾] B§2.2 × A§3.3 · 大盘首屏的全局聚合,注册表元数据不够 · 重要**
B 顶栏要"3 件待你处理 · 12 在跑 · 4 机在线 · 数据 40s 前"——跨机会话级状态聚合;A 的注册表只有 machine→端口、能力清单、最后心跳(§3.3"元数据 only")。若走数据面逐机逐 harness 拉取,首屏 = N×M 次隧道往返,离线机各贡献一个超时,S2 的"≤5s 确认没我的事"做不到;A §0.2 说心跳"带元数据与脱敏标题"(暗示会话级上报)与 §3.3"只带元数据"自相矛盾,上报粒度内部就没定。
· 建议改法:控制面心跳升级为「会话级状态摘要(9 态 + 计数 + 脱敏标题)」并写进 A §3.3——这是大盘成立的必要条件,代价是注册表从机器级变会话级,属于对 A 的真实修订。
