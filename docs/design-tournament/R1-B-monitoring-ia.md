# R1-B · 多项目并行监控大盘 · 信息架构（监控效率路）

> 设计锦标赛 Round 1 · B 路（监控效率）。
> 边界声明：本路只负责「注意力的仲裁与呈现」。跨 harness 事件采集归技术适配层路；P0 的触达渠道/频控归通知协议路。本路的输出 = 分级语义 + 一屏信息架构 + 状态机契约 + 验证脚本。
> 方法：全部结论从「注意力经济学」第一性推导，每条给出可机器执行的判定，不放空话。
> 承继：fleet-cockpit v2（注意力预算/数据谱系/深链不割裂）、SPEC-multisession（状态派生非自报/心跳 TTL）、acceptance-walkthrough 维度库、kimi-remote 现有 H5（会话列表/详情/PWA）。

## 1. 本质命题还原：监控 = 注意力仲裁，不是系统展示

### 1.1 第一性推导

王佐多项目并行时，监控的本质命题只有一句：

> **此刻，我这一分钟的注意力应该花在哪？——还是哪都不用花？**

推导链：

1. **供给**：王佐给监控的时间是碎片（通勤/间隙/睡前），单次打开大盘的真实预算 ≈ 30–90 秒。手机是唯一入口（飞书+H5，D-orca-scope 已实证）。
2. **需求**：几十会话 × 多机 × 多 harness，每小时产生上百条状态变化。
3. **矛盾**：信号量 ≫ 注意力供给。因此大盘的第一功能不是「展示」，是**删除**——把 99% 的变化判定为「不用看」，剩下的严格分级、按代价排序。
4. **打开场景分布**：80% 的打开是为了确认「没有事需要我」（安心型），20% 是来处理事（介入型）。
5. 所以大盘的北极星指标 = **安心成本**：王佐为了确信「一切正常」所付出的打开次数 × 停留时长。**这是一个用得越少越成功的产品**——大盘的胜利是王佐每天少打开 10 次，不是多停留 10 分钟。

推论（设计硬约束）：**一屏内「需要注意力」的条目 ≤ 7±2**（工作记忆上限）。超出说明分级或聚合失败，正确动作是按项目/会话折叠，不是加滚动。

### 1.2 「需要人介入」的严格三级判定

判定的总标准只有一条，全部从它派生：

> **「我不动，它就不动；我晚动，它就晚好；我不动，它就死掉或造成损失。」→ P0。**

| 级 | 定义 | 精确判定条件（事件级，机器可判） | 王佐该做什么 |
|---|---|---|---|
| **P0 必须看** | 阻塞型：系统无法单方面继续，王佐的动作是恢复进展的充分必要条件 | ① 存在未决 approval 请求且会话此后无新事件（待审批）；② 存在未决 question（待回答）；③ 任务/进程终态非零退出**且**无自动重试进行中（失败）；④ 有未完成任务 + 最后事件超 TTL 30min（僵死，承心跳协议）；⑤ 红线授权请求（钱/生产/凭据/对外发布） | 立即处理；排序按**阻塞时长**降序（等得越久越在烧等待成本） |
| **P1 可以看** | 非阻塞型：王佐不动，系统照样演进；但信息进入心智模型有后续价值 | ① 任务达终态（有 receipt）且未读（完成未读）；② prompt 已入队未执行（排队）；③ 异常已自愈（重试成功/隧道自动重连）——只留记录；④ 长任务里程碑（构建/测试通过） | 碎片时间扫一眼，3 秒内可关掉；可批量已读 |
| **P2 不用看** | 与「要不要动作」无关的背景态 | ① 运行中（busy 是默认态不是信息——在跑恰恰说明不需要你）；② 空闲（已完成已读/无任务）；③ 预期内的排队时长；④ token 消耗、流式输出本身 | 不放上屏；聚合成一行摘要证明「大家还活着」 |

反空话检验：每条判定都能写成 `if` 语句，不依赖「重要」「异常」这类主观词。注意 ③④ 的限定语（「无自动重试进行中」「有未完成任务」）——没有它们，瞬时重启和正常完成的空闲会话会制造大量假 P0，毁掉整个分级（误报率是比漏报率更隐蔽的杀手：它训练王佐无视大盘）。

**排序键 = 阻塞时长 × 阻塞代价，不是 recency。** 最新发生的事≠最该处理的事；挂了 47 分钟的审批比 30 秒前的审批优先。

### 1.3 与另两路的接口

- → 通知协议路：消费本分级。只有 P0 配打断（推送/强提醒）；P1 只进 digest；P2 永不触达。
- → 技术适配层路：消费 §3 状态机契约。本路不管事件怎么采，只要求适配层交付「未决 approval 存在性」「最后事件时间戳」「终态+receipt」这几个判据字段。

## 2. 大盘信息架构：异常优先的收件箱，不是状态博物馆

### 2.1 范式选择（推导，非偏好）

王佐打开大盘的第一问是「有没有事需要我」，不是「项目 X 怎么样了」（那是复盘场景，归 cockpit 项目视图）。所以：

- 第一排序维度 = **介入紧迫度**（P0 → P1 → P2 折叠），第二维度 = 项目（同级内聚合）。
- **否决状态矩阵**（运行/排队/失败各一区平铺）：它会让 30 条「运行中」绿点永远占据首屏，把唯一的「失败 1」淹死——状态博物馆，见 §5 反模式 1。
- **否决项目看板做首屏**：项目是好的聚合键、坏的首问。
- **否决时间线做首屏**：流水账不回答「要不要我」。

结论：范式 = **异常优先的收件箱式分组列表 + 钻取**。与 fleet-cockpit v2 手机首页信息预算对齐（≤3 件待处理 + ≤1 条异常 + 1 行新鲜度）；本盘作为 cockpit 的「会话监控」视图，遵守同一预算。

### 2.2 一屏结构（手机 390px，自上而下）

```
┌────────────────────────────────────────────┐
│ 顶栏: 3 件待你处理 · 12 在跑 · 4 机在线 · 数据 40s 前 │ ← 安心摘要+新鲜度, 5% 高
├────────────────────────────────────────────┤
│ ① 需要你 (P0)                        🔴 3   │ ← ~55% 高, 大盘存在的理由
│   按阻塞时长降序; 每行=一个可决断项, 行内可处理  │
├────────────────────────────────────────────┤
│ ② 待过目 (P1)                        🟢 4   │ ← ~25%; 完成未读为主
│   按完成时间降序; 右上角「全部已读」           │
├────────────────────────────────────────────┤
│ ③ 在跑: 12 个会话运行中 · 4 排队 · 3 机在线  › │ ← P2 折叠为一行摘要, ~10%
├────────────────────────────────────────────┤
│ ④ 异常 (仅在有异常时出现): M2 隧道断开 · 数据源陈旧 │ ← 不占默认预算
└────────────────────────────────────────────┘
```

- P0 > 3 条时，第 4 条起折叠为「还有 2 件待你处理」一行，展开不占别人预算。
- ③ 摘要行的存在理由：让王佐**不用点进去**就能确认「舰队还活着」。它替代了 20 行绿点。
- **空态**：P0 为空时显示一行「✓ 没有需要你介入的事」——安心必须显式表达；空白 = 用户分不清是没数据还是没加载（验收走查 E 维度）。

### 2.3 列表行信息密度设计（每行放什么、不放什么）

**P0 行 —— 行 = 决断单元。** 设计目标：80% 的情况不点进去就能决断。

```
🔴 12m   kimi-remote · M5 · Kimi Code
         待审批: 允许写入 docs/design.md?
         [批准]  [拒绝]  [看上下文 ›]
```

- **放**：① 阻塞时长（最醒目，相对时间，分钟级）；② 项目名 · 机器 · harness（一行小字，同项目跨机时是定位关键）；③ 等什么——审批内容/问题摘要/失败原因，**一句大白话**（走查 K 维度：禁止甩 `upstream unreachable` 式原文）；④ 行内最小动作（批/拒/一句话回答）。
- **不放**：消息数、token 数、会话 ID、完整 cwd、最后消息预览、绝对时间戳——全是详情页的事，放上来就是稀释信噪比。

**P1 行 —— 行 = 验收单。**

```
🟢  2h 前完成   zaios 舰队 · 总控
    蜂群派发 6 任务: 5 成功 · 1 重试中
```

- **放**：完成时间 + 一句话**结果摘要**（注意：是任务目标的收口，不是最后一条消息——「已回复 3 封邮件」而非「邮件已发送，请问还…」）。
- **不放**：操作按钮（除右滑已读/点开验收）。

**P2 行（③ 展开后）—— 行 = 存活证明。**

```
⚪  客户项目A · M2 · Codex     已运行 47m · 最后事件 2m 前
```

- 只放身份 + 已跑时长 + 最后事件时间（判僵死的原料）。**零动作、零红点。**

### 2.4 钻取路径（点击预算）

| 目标 | 路径 | 点击数 |
|---|---|---|
| 处理一个 P0 | 点行 →（进行情对应位置）→ 点批准/回答 | ≤2 |
| 验收一个 P1 | 点行 → 详情页（自动滚到结果处） | 1 |
| 任意会话的任意细节 | 大盘行 → 会话详情（流式/审批/排队/回执，已有）→ 全档历史 | ≤3 |
| 从飞书到任意层 | 深链直达（`/s/<id>` 已有稳定 URL） | 1 |

硬规则（承 v2「下钻不制造割裂」）：
- 每一层有稳定服务端 URL；状态存服务端/hash，不靠浏览器内存——飞书 WebView 会被回收，杀进程重进必须回到原滚动位置。
- 「看材料 → 做决定 → 看回执」是一条任务流，不是三次下钻返回。
- 返回键永远回到大盘原位置、原折叠状态。

### 2.5 手机一屏优先级（首屏 100vh 信息预算分配）

1. 顶栏安心摘要（5%）——回答「要不要继续往下看」。
2. P0 前 3 条（~55%）——大盘存在的理由。
3. P1 前 2 条 + 全部已读（~25%）。
4. P2 摘要行（~10%）——存活证明。
5. 异常条（条件出现）。

PC 端允许并排铺开（P0 列 + P1 列 + 在跑网格），但**手机不铺开**（v2 已定：纵排多窗格=被否定的一页流）。

## 3. 状态模型：跨 harness 统一状态机

### 3.1 总原则

- **状态派生非自报**（承 SPEC-multisession）：由事件推导，不信 harness 自述字段。教训已发生：`pending_interaction:"none"` 是字符串真值，被误显示为「待处理」（走查 C 维度）。判据必须落在「未决请求事件是否存在」，不是字段字符串。
- **缺失显示 unknown，禁默认 green**（承 v2 数据谱系）：机器离线时其上的会话不是「空闲」，是「未知」。
- **状态互斥**：任一时刻一个会话恰好处于一个态。UI 不许一行出现两个状态点（§5 反模式 8）。
- 状态迁移带时间戳入账（这是 §4 时间视图和「阻塞时长」排序的数据源）。

### 3.2 状态机（9 态）

| 态 | 派生条件（事件级） | 注意力级 | 迁移 |
|---|---|---|---|
| `blocked_approval` 待审批 | 存在未决 approval 事件，且其后无会话事件 | **P0** | 批/拒 → `running` |
| `blocked_question` 待回答 | 存在未决 question 事件，且其后无会话事件 | **P0** | 回答 → `running` |
| `failed` 失败 | 终态非零退出，且无自动重试进行中 | **P0** | 重试 → `running`；放弃 → `idle` |
| `stalled` 僵死 | 有未完成任务 + 最后事件距今 > TTL(30min) | **P0** | 新事件 → `running`；确认死亡 → `failed` |
| `done_unread` 完成未读 | 达终态（有 receipt）+ 未达清除条件 | **P1** | 清除 → `idle`（§3.3） |
| `queued` 排队 | prompt 入队事件存在、无 turn_start | P1（超阈值 → `stalled`） | turn_start → `running` |
| `running` 运行中 | 最近事件窗内有 turn/工具事件 | P2 | → blocked_*/done_unread/failed |
| `idle` 空闲 | 无未完成任务、无未读 | P2（不上屏） | 新 prompt → `queued`/`running` |
| `unknown` 未知 | 数据源离线/判据字段缺失 | 特殊（灰 ?） | 数据恢复 → 重派生 |

关键决策及理由：

- **审批与回答拆成两态**：动作不同（批/拒 vs 输入内容），行内按钮不同，合并态会让行内动作二义。现有 H5 的「待处理」单态（`app.js:293`）要拆。
- **`stalled` 独立成态**：它是最隐蔽的注意力杀手——「以为在跑其实死了」，现有 H5 无此态，王佐只能靠猜（走查 A 维度真实反例）。TTL 承心跳协议 30min；测试时可压缩。
- **`failed` 防瞬时噪音**：必须带「无自动重试进行中」限定；被 launchd 拉起的瞬时崩溃属「已自愈」，只进 P1 记录，绝不响 P0。
- **`unknown` 是一等公民**：隧道断开时，该机上全部会话置 `unknown` 并计入 §2.2 ④ 异常区——不许沿用最后已知状态装活。

### 3.3 「完成未读」的清除逻辑

进入：会话达任务终态（receipt 事件）→ 置 unread。
清除（三选一，满足即清）：

1. **显式**：P1 行右滑「已读」，或「全部已读」。
2. **隐式**：进入该会话详情停留 ≥3 秒**且**滚到最后一条消息（看到了结果才算）。游标存**服务端**（承 ROADMAP 的 last_seq/已读游标计划，但必须服务端化——换设备、飞书 WebView 重载都不清零）。
3. **过期**：>72h 未读自动降为已读。理由：三天不看 = 不重要；留着只会培养「未读数不可信」的坏习惯，毁掉整个未读体系。过期清除的项进本周视图可考古，不是删除。

防误清硬规则：
- 进详情 1 秒就退出 ≠ 已读（手滑）。
- 「全部已读」**只对 P1 生效，永不清 P0**——待审批不能靠「全部已读」消失，这是安全底线。
- 已读游标是**每会话独立**的，不是全局 watermark——否则旧会话的已读动作会误清其它会话的新完成。

## 4. 时间维度：三个视图回答三个不同的问题

| 视图 | 回答的问题 | 内容 | 使用场景 |
|---|---|---|---|
| **现在 Now**（默认） | 此刻要不要我？ | §2.2 那一屏（P0+P1+P2 摘要+异常） | 90% 的打开 |
| **今天 Today** | 我今天推动了什么、漏了什么？ | 今日状态迁移的**收成清单**：完成 8 · 失败 1 · 你处理了 5 个审批 · 当前 12 在跑；按项目聚合 | 睡前 30 秒复盘 |
| **本周 Week** | 有什么趋势和系统性问题？ | 吞吐（完成/天）、失败率、**阻塞时长分布**（哪台机/哪个 harness 最常卡住我）、注意力开销（每天推我几件 P0） | 发现「M2 的会话老僵死 → 该修机器，不是每次手工处理」 |

- 「本周」是唯一允许出现统计图表的视图；「现在」视图零图表（图表是分析工具，不是仲裁工具）。
- 视图切换是顶部单指段控件（Now/Today/Week），默认永远落在 Now。

### 4.1 「我离开这段时间发生了什么」一键回溯（Away 摘要）

- **入口**：顶栏常驻按钮「我离开的 3h 25m · 看变化」。离开时长基于**服务端记录的上次打开时间**（不是 localStorage——换设备也对）。离开 <10 分钟不显示此按钮（没有「离开」，防止变成习惯性多点一次）。
- **内容 = 净变化差量摘要，不是事件流水**。按注意力级预分层：
  1. **仍未处理的 P0**（发生过且还挂着）→ 置顶，行内可直接处理；
  2. **已了结的**（完成了/失败自愈了/超时取消了）→ 一行一条，只读；
  3. **持续在跑的**（离开前在跑、现在还在跑）→ 聚合一行「4 个会话持续运行」。
- **压缩硬规则**：任意时长的事件量压缩到 ≤7 条净变化，同类折叠——「kimi-remote 完成 3 次迭代」是一行，不是三行。30 条原始事件进，≤7 条净变化出。
- 原则：回溯回答「世界在我离开时的净变化」。事件流水账是日志，日志不上屏（§5 反模式 5）。

## 5. 反模式清单（大盘不许做什么）

| # | 不许做 | 为什么（依据） |
|---|---|---|
| 1 | 按状态分 tab/分区平铺（运行 20 / 空闲 30 / 失败 1 并列） | 状态博物馆：唯一的「失败 1」被绿点海洋淹死。异常优先是唯一合法排序 |
| 2 | 显示「运行中」的进度条/百分比/token 流 | LLM 任务没有真进度；假进度比没进度更糟（走查 C：假状态比没状态更糟） |
| 3 | 给 P2 级信息挂红点/未读数（「某会话新消息 +3」） | 运行中的会话产生消息是本职。挂红点 = 训练王佐无视红点。红点只属于 P0/P1 |
| 4 | 大屏崇拜：常亮电视墙、几十宫格会话缩略图 | 王佐的监控入口是手机碎片时间；大屏是工程虚荣，服务的是作者不是用户 |
| 5 | 把原始事件流当信息（「10:23 Read / 10:24 Edit」流水） | 那是日志不是监控。上屏最小单位 = 状态迁移 + 一句人话摘要（走查 K） |
| 6 | 高频无差别轮询 | 列表轮询必须 `document.hidden` 熔断（走查 J 已固化）；状态变化走 WS 推送，轮询只是断线兜底 |
| 7 | 默认 green：数据陈旧/源离线时沿用最后成功状态 | v2 数据谱系硬标准：缺失显示 unknown。华佗 stale 告警已有，必须上屏 |
| 8 | 一行多义：同行出现「运行中」+「待审批」两个状态点 | 状态机互斥，UI 必须互斥呈现；让人猜哪个真 = 状态不可见（走查 A） |
| 9 | 把大盘做成操作中心（派活/配置/翻历史全塞进来） | 大盘是仲裁注意力的地方，不是第二个工作台。行内只放处理 P0 的最小动作；其余走详情页/cockpit（v2 已定边界） |
| 10 | 按 harness 区分视觉权重（大卡/小卡/不同色） | harness 只是行内小字标签。王佐关心「什么事」，不关心「哪个工具」 |
| 11 | 「全部会话」做默认视图 | 默认视图永远 = 需要你 + 待过目。「全部」藏进筛选器，主动找才有 |
| 12 | P1 级信息用声音/震动强提醒 | P1 不配打断。打断权只属于 P0，渠道与频控归通知协议路 |
| 13 | 会话标题原文直出到列表 | 走查 M：标题可能含客户密码片段/内部代号。上屏标题需脱敏白名单（这是真实踩过的坑） |

## 6. 验证方式：任务脚本实测

总原则承 acceptance-walkthrough：真机 390 视口、真实账号、真实数据；每个脚本布置后计时；数据类断言必须拉接口原始响应核对（UI 显示 vs 字段真值），不许「应该对了」。

| 脚本 | 布置 | 任务 | 通过标准 | 命中维度 |
|---|---|---|---|---|
| **S1 30 秒找出全部待介入**（核心） | 12 会话：2 待审批 + 1 待回答 + 1 失败 + 8 正常 | 打开大盘，说出全部 4 个待介入项，并行内处理 1 个审批 | ≤30s；零漏报零误报；处理 ≤2 次点击 | G/H |
| **S2 安心成本**（80% 场景） | 全部正常，零 P0/P1 | 确认「没我的事」 | ≤5s；不滚动不点开；显式空态「没有需要你介入的事」+ 新鲜度可见 | A/E |
| **S3 离开回溯** | 离开 2h，期间：2 完成、1 失败自愈、1 审批仍挂、3 持续在跑 | 60s 内复述净变化并处理挂着的审批 | Away 摘要 ≤7 条；净变化零遗漏；未处理 P0 置顶 | G |
| **S4 僵死与假活** | 杀 1 个会话进程但保留界面；断 1 台机隧道 | 观察大盘 | TTL 内该会话转 `stalled` 进 P0；断机全部会话转 `unknown`（不许绿）；异常区出现该机 | A/C |
| **S5 完成未读清除** | 3 会话完成：a) 右滑已读 b) 详情停 5s 滚到底 c) 详情 1s 退出 | 执行三种动作 | a/b 清除、c 不清；「全部已读」后 P0 纹丝不动 | B/H |
| **S6 钻取深度** | 任意指定会话 | 从大盘到该会话最后一条消息 | ≤3 次点击；杀 WebView 重进后服务端状态恢复到原位 | H/I |
| **S7 信噪比压测** | 50 会话 × 4 机 × 3 harness，只有 2 个真 P0 | 打开大盘 | 首屏（不滚动）两个 P0 都可见；首屏 P2 条目数 = 0（折叠为摘要）；单屏总条目 ≤9 | G |

持续度量（上线后每周）：
- **安心成本** = 每日打开次数 × 平均停留时长 → 目标：次数降、时长 ≤30s。
- **P0 响应时延** = 阻塞发生 → 处理 → 目标：下降。
- **误报率/漏报率**：每周抽查 20 条状态迁移，UI vs 接口真值核对（走查 C 方法）。任一 >5% 即回炉 §1.2 判定条件。
